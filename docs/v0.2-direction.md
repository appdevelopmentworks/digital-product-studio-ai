## 2. オープンクエスチョン Q1〜Q6 の確定回答

### Q1: 段階展開戦略

**決定: 採用**

```
v0.2 (Production-ready core):  20エージェント
   - Tier 0: 1
   - Tier 1: 5(全 Practice Director)
   - Tier 2: 6(主要 Discipline Lead のみ)
   - Tier 3: 6(Critical specialists のみ)
   - Tech Stack: 2(nextjs / wordpress のみ)

v0.3 (Expansion):              30エージェント
   - Product Practice の specialist 拡充(SaaS開発系)
   - Tech Stack: saas-stack, headless-cms, astro 追加

v0.4 (Full):                   38+エージェント
   - EC・i18n 等の追加
   - Tech Stack を全領域カバー
```

**各段階で1案件回して検証してから次に進む**(段階展開の必須条件)。

### Q2: 自社プロダクト開発も対象

**決定: 含める**

AILEAP 自社プロダクト開発も digital-product-studio-ai の対象とする。自社案件は仮想的な「内部クライアント = AILEAP」として扱い、受託案件と同じワークフローで進行する。

これにより:
- ドッグフーディングで本組織自体の品質が向上
- AILEAP のプロダクトラインが本組織の事例として `/proposal-deck` で活用可能
- 既存資産(MeetingAI、AILEAP website 等)を本組織でメンテナンス可能

### Q3: 動作環境

**決定: Windows + WSL2 をプライマリ環境**

```
プライマリ:  Windows 11 + WSL2 + Git Bash
セカンダリ:  macOS / Linux(best-effort、動作未保証)
```

理由:
- Shin の作業環境が Windows(C:\Users\hartm\)
- bash hooks は WSL2 / Git Bash 前提で実装
- PowerShell 版は v0.4 以降で検討(現在は対象外)

### Q4: 法務テンプレート

**決定: v0.2 でひな型作成、ただし弁護士確認注意書きを強制**

法務3テンプレート(privacy-policy / terms / tokushoho)は v0.2 で作成するが、テンプレート冒頭に必ず以下を強制挿入する:

```markdown
> **⚠️ 法務確認必須**
> 本テンプレートはひな型であり、AILEAP は法的助言を行いません。
> 本番運用前に必ず弁護士による確認を受けてください。
> 本テンプレートの使用に起因する一切の損害について、
> AILEAP は責任を負いません。
```

実装方法:
- テンプレート生成時に上記ヘッダーを必ず挿入
- `legal-pages-check.sh` hook がこのヘッダーの存在も検証
- 該当ヘッダー未挿入のままデプロイされる事故を防ぐ

### Q5: 多言語対応(日英中韓4言語)

**決定: v0.2 で前倒し採用**

理由:
- Shin が日英中韓のネイティブまたは高度話者
- AILEAP の差別化要素として位置づけ可能
- 海外 SME クライアント開拓の入り口になる

v0.2 での実装範囲:
- `localization-specialist` を v0.2 から Tech Stack Specialist に追加
- `/i18n-strategy` スキルを v0.2 で実装
- `/localization-audit` スキルを v0.2 で実装
- next-intl / Astro i18n / WordPress Polylang の3パターンに対応
- ただし v0.2 段階展開の20エージェント枠を圧迫しない範囲で実装(下記4参照)

### Q6: リポジトリ統合

**決定: 別リポジトリ維持**

apex-consulting-ai / digital-product-studio-ai / web-marketing-ai-org は独立リポジトリとして運用。理由は gap-analysis Section 9 Q6 参照。

### 3.3 重要: エージェント出力の言語ルール

エージェントプロンプトは英語で記述するが、**出力時はユーザーの母国語に合わせる**ことを必ず明記する。

全エージェントプロンプトの末尾に必ず以下を含める:

```markdown
## Output Language Policy

**ALWAYS respond in the user's native language.** Detection rules:

- Internal team communication (with Shin / AILEAP): **Japanese**
- Client deliverables (proposals, requirements docs, presentations):
  Match the client's language. Default to **Japanese** for domestic SME
  clients in Japan unless otherwise specified.
- Multi-language sites (i18n projects): Generate content in all target
  languages as specified in the project's i18n configuration.
- Technical artifacts read only by other agents (intermediate YAML,
  internal logs): English is acceptable.

When in doubt, default to **Japanese**.
```

## 4. v0.2 で必ず解消する Critical 12件

gap-analysis-v0.1.md Section 6.1 で特定された12件。v0.2 でこれら全てを解消する。

### Phase 1: 設計再構築

**F-C1: 段階展開戦略の採用**(Q1 で確定済み)
- v0.2 は20エージェントで開始
- v0.3 は30、v0.4 は38+
- 各段階で1案件検証を必須化

**CO-C1: エージェント数の整合性**
- v0.1 で混在した「37 vs 38」を解消
- v0.2 では明確に「20エージェント」と一意に記載

**D-C1: 差別化軸の明文化**
- 新規セクション「Differentiation Axis」を v0.2 に追加
- 8軸の比較表(速度・価格・技術選定・GEO・修正対応・コンサル統合・多言語・a11y)
- 全エージェントがこの軸を意識する形で設計

### Phase 2: 顧客対応プロセス強化

**C-1: クライアント合意形成プロセス**
- 新規エージェント `approval-tracker`(Sonnet・Delivery 配下)
- 新規スキル `/approval-request`、`/approval-record`、`/approval-status`
- 新規 hook `pre-deploy-approval-check.sh`

**C-2: 議事録・コミュニケーション履歴管理**
- ディレクトリ構造拡張: `projects/{id}/00-engagement/meetings/`、`correspondence/`、`decisions.yaml`
- 新規スキル `/meeting-minutes`、`/decision-log`

**C-3: 価格戦略・収益モデル**
- 新規ドキュメント `docs/pricing-strategy.md`(日本語)
- `commercial-manager` プロンプト強化(Time&Material / Fixed / Retainer の3パターン提案を必須化)
- 新規スキル `/retainer-design`、`/value-pricing-analysis`

**C-4: アセット受領管理プロセス**
- 新規エージェント `asset-coordinator`(Sonnet・Delivery 配下)
- 新規スキル `/asset-checklist`、`/asset-status`
- ディレクトリ構造拡張: `projects/{id}/01-discovery/assets-received/`、`assets-required.yaml`

### Phase 3: 責任分界の精緻化

**B-C1: SEO/GEO の段階別境界**
- 4段階に分割: 設計 / 実装 / 公開後30日検証 / 継続改善
- 設計〜検証は digital-product-studio-ai
- 継続改善は WMAO
- ハンドオフプロトコル `/handoff-to-marketing` に「30日経過」を明記

**B-C2: 公開時の初期コンテンツ制作の境界**
- ローンチ時の「初期コンテンツ5-10本」は digital-product-studio-ai 側で制作
- 11本目以降は WMAO 引継ぎ
- ハンドオフプロトコルで「初期コンテンツ件数」を明記

**B-C3: 制作系/開発系のモード切替**
- エージェントプロンプトに「制作モード/開発モード」のスイッチ
- `/team-*` コマンド起動時に自動切替

### Phase 4: 実現可能性確保

**F-C2: 並列案件数の現実的制約**
- 同時アクティブ案件数の上限を **2-3案件** と明記
- 案件切替時のセッション分離プロトコル追加
- `studio-director` に「案件切替時の context flush」責任を明記

**CO-C2: 案件タイプとチーム編成の整合**
- v0.2 では「汎用 `/team-assemble` + 案件タイプ引数」方式を採用
- 個別の `/team-corporate-site` 等は v0.2 では3個に絞る(corporate / landing-page / mediasite)
- 残り8タイプは v0.3 / v0.4 で段階拡張

## 12. 注意事項

### 12.1 段階展開の原則

v0.2 は20エージェントで完結させる。誘惑に負けて v0.3 / v0.4 のエージェントを先取り実装しない。理由は段階ごとに実案件で検証する設計のため。

### 12.2 言語ポリシーの一貫性

ドキュメントの一部だけ英語、一部だけ日本語のような混在を避ける。本書 Section 3 のルールに従って厳格に分離する。

### 12.3 法務責任の明確化

法務テンプレートには必ず「弁護士確認必須」ヘッダーを挿入する。AI が法的助言を行ったと誤解されないよう、責任分界を明示する。

### 12.4 自社プロダクトの特殊扱い

AILEAP 自社案件は通常案件と同じワークフローで進めるが、`commercial-manager` は内部見積を生成する(請求書は発行しない)。

### 12.5 多言語対応の実装範囲

Q5 で前倒し採用したが、v0.2 では `/i18n-strategy` スキルと `localization-specialist` のみ実装。全テンプレートの4言語化は v0.3 以降。
