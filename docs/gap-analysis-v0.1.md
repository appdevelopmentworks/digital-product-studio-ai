# digital-product-studio-ai Gap 分析レポート v0.1

**対象文書**: `docs/requirements-v0.1.md`
**分析日**: 2026-04-27
**分析者**: Claude(自己批判モード)
**目的**: v0.1 の弱点を網羅的に洗い出し、v0.2 改訂への入力とする
**参照前例**: apex-consulting-ai は同手法で v0.1 (55%) → v0.2 (88% MBB readiness) に到達

## 0. エグゼクティブサマリー

### 総合評価

| 観点 | スコア | 主要ギャップ件数 |
|---|---|---|
| 完備性(Completeness) | 72/100 | Critical: 4 / High: 6 / Medium: 8 |
| 責任分界(Boundary clarity) | 68/100 | Critical: 3 / High: 4 / Medium: 5 |
| 実現可能性(Feasibility) | 75/100 | Critical: 2 / High: 3 / Medium: 6 |
| 整合性(Consistency) | 70/100 | Critical: 2 / High: 4 / Medium: 5 |
| 差別化(Differentiation) | 78/100 | Critical: 1 / High: 3 / Medium: 4 |
| **総合(参考: 業界標準スタジオ readiness)** | **73/100** | **Critical: 12 / High: 20 / Medium: 28** |

### 結論

v0.1 は「ゲームスタジオを Web に翻訳した骨格」としては成立しているが、**Web/デジタル業界特有の運用知識の翻訳が浅く、実運用に投入すると初回案件で4〜5箇所が破綻する状態**である。とくに Critical 12件のうち以下3件は v0.2 で必ず解消する必要がある。

1. **顧客との合意形成プロセスがエージェント設計に組み込まれていない**(Section 1 詳述)
2. **計測・データ・分析プラクティスが Engineering 配下に押し込まれて独立性がない**(Section 2 詳述)
3. **AI Native 制作会社としての差別化要素が要件定義に明文化されていない**(Section 5 詳述)

apex の v0.1→v0.2 は55→88(+33pt)を達成。本分析の Critical/High すべてを解消すれば 73→90 程度(+17pt)が見込める。

## 1. 完備性(Completeness)— 72/100

### 1.1 Critical Gaps(4件)

#### C-1: クライアント合意形成プロセスの欠落

**現状**: 要件定義書の中で「クライアント承認」という言葉は Section 11(Collaborative-Not-Autonomous)に1度出てくるのみ。誰が、いつ、何を承認するのか、承認後に何が起こるのかが未定義。

**ギャップ**: Web 制作・開発の事故の8割は「言った言わない」問題。承認プロセスの構造化なしには、AI が生成した提案を Shin が口頭で OK したのか、クライアントがメールで OK したのか、議事録に残っているのか、が混線する。

**v0.2 改訂提案**:

新規エージェント追加:
- `approval-tracker`(Sonnet・Delivery Practice 配下)— 全承認イベントを `projects/{id}/approvals.yaml` に記録

新規スキル追加:
- `/approval-request` — クライアントへの承認依頼書(メール下書き含む)生成
- `/approval-record` — 承認結果の記録
- `/approval-status` — 案件全体の承認状況可視化

新規 hook:
- `pre-deploy-approval-check.sh` — 必須承認(デザイン承認・公開承認)が揃っていない場合デプロイをブロック

**優先度**: Critical / **工数**: 中

---

#### C-2: 議事録・コミュニケーション履歴管理の欠落

**現状**: 要件定義書にミーティング議事録や Slack ログをどう扱うかの記述なし。

**ギャップ**: クライアントとの全コミュニケーションが「Shin の頭の中」に閉じ込められると、AI エージェントは案件文脈を把握できない。逆にクライアント発話の文字起こしを案件ディレクトリに置けば、全エージェントが文脈を共有できる。

**v0.2 改訂提案**:

ディレクトリ構造追加:
```
projects/{id}/
├── 00-engagement/
│   ├── meetings/         議事録(日付_キックオフ.md など)
│   ├── correspondence/   メール・チャットの抜粋
│   └── decisions.yaml    意思決定ログ
```

新規スキル:
- `/meeting-minutes` — 録音・メモから構造化議事録生成
- `/decision-log` — 重要意思決定をエスカレーションパス付きで記録

**優先度**: Critical / **工数**: 小

---

#### C-3: 価格戦略・収益モデルの設計が欠落

**現状**: `commercial-manager` エージェントと `/estimate` スキルは存在するが、見積の「単価設計」「リテイナー」「成果報酬」「ストック収益化」の戦略が要件に入っていない。

**ギャップ**: AILEAP の経営目標(memory 上の「ambitious annual revenue target」)を達成するには、案件単価設計と LTV 戦略が要件レベルで定義されている必要がある。「制作」だけで売っていると digital-product-studio-ai は労働集約に陥る。

**v0.2 改訂提案**:

新規ドキュメント:
- `docs/pricing-strategy.md` — 案件タイプ別の単価レンジ、リテイナー設計、成果報酬の発動条件

エージェント `commercial-manager` のプロンプト強化:
- Time & Material / Fixed Price / Retainer の3パターン提案を必須化
- 月額保守(Retainer)を必ず提案する強制ルール

新規スキル:
- `/retainer-design` — 月額保守契約設計
- `/value-pricing-analysis` — 成果連動型課金の妥当性分析

**優先度**: Critical / **工数**: 中

---

#### C-4: クライアント側のアセット受領・管理プロセスの欠落

**現状**: クライアントから受領するアセット(ロゴ・写真・原稿・既存サイトデータ・各種アカウント情報)の管理仕様が未定義。

**ギャップ**: 業界実態として「クライアントからアセットが届かない」が公開遅延の最大要因。受領状況をエージェントが追跡できないと、待ちフェーズが永遠に続く。

**v0.2 改訂提案**:

新規エージェント:
- `asset-coordinator`(Sonnet・Delivery Practice 配下)— アセット受領状況を追跡、未提出アセットをリマインド

新規スキル:
- `/asset-checklist` — 案件タイプに応じた必要アセット一覧生成
- `/asset-status` — 受領状況可視化、未受領のリマインドメール下書き生成

ディレクトリ構造追加:
```
projects/{id}/01-discovery/
├── assets-received/      受領済みアセット
├── assets-required.yaml  必要アセットチェックリスト
└── credentials/          認証情報(暗号化前提)
```

**優先度**: Critical / **工数**: 小

### 1.2 High Gaps(6件)

#### H-1: ユーザビリティテスト・ユーザーリサーチの位置づけが弱い

**優先度**: High / **工数**: 中

#### H-2: データ分析プラクティスが薄い

**現状**: `measurement-architect` が GA4/GTM 設計を担うが、「公開後のデータ分析」「ファネル分析」「ヒートマップ運用」が誰の担当か曖昧。WMAO に丸投げするのは短絡的(WMAO はマーケ運用が主軸)。

## 2. 責任分界(Boundary clarity)— 68/100

### 2.1 Critical Gaps(3件)

#### B-C1: SEO/GEO の「設計」と「運用」の境界が曖昧

**現状**: 要件定義書 Section 1.3 で「公開後の継続SEO・GEO改善運用は WMAO」と書きつつ、Section 4.4 で `seo-geo-strategist` が digital-product-studio-ai 側にいる。

**問題**: 「公開直後30日のSEO効果検証」は誰がやるのか?「初動キーワード順位モニタリング」は?境界に grey zone が広い。

**v0.2 改訂提案**: SEO/GEO の責任を3段階に分割して明文化

| 段階 | 担当組織 | 期間 |
|---|---|---|
| 設計(キーワード戦略・サイト構造・構造化データ) | digital-product-studio-ai | 設計フェーズ |
| 実装(メタタグ・構造化データ・llms.txt・sitemap) | digital-product-studio-ai | 実装フェーズ |
| 公開直後検証(30日間の初動分析) | digital-product-studio-ai | 公開後30日 |
| 継続改善(コンテンツSEO・被リンク・順位改善) | WMAO | 30日以降 |

ハンドオフプロトコル(2.1.2)に「30日経過」を明記、それまでの検証データを WMAO に引き渡す。

**優先度**: Critical / **工数**: 小

---

#### B-C2: コンテンツマーケティングの初回コンテンツは誰が作るか不明確

**現状**: 公開時の「初期コンテンツ20本」は誰が作るか未定義。`copywriter` が制作するのか、WMAO が制作するのか。

**問題**: 公開時に空のメディアサイトを納品するのは現実的でない。一方 WMAO の継続コンテンツマーケと混線する。

**v0.2 改訂提案**:
- ローンチ時の「初期コンテンツセット(目安: 5-10本)」は digital-product-studio-ai 側で `copywriter` + `content-strategy-lead` が制作
- 11本目以降は WMAO に引継ぎ
- ハンドオフプロトコルで「初期コンテンツ件数」を明記

**優先度**: Critical / **工数**: 小

---

#### B-C3: 「Web 制作」と「Web 開発」のチーム動員ルールが暗黙のまま

**現状**: 案件タイプ別ワークフローで `/team-corporate-site`, `/team-saas-mvp` 等が示されているが、「制作系・開発系・ハイブリッド」の分類がエージェント・スキル設計に明示的に反映されていない。例えば `frontend-engineer` は両方に使われるが、要求される技能が異なる(制作系=デザイン再現・LP最適化、開発系=状態管理・性能チューニング)。

- スタックスペシャリスト呼び出しもモードに応じて切替

**優先度**: Critical / **工数**: 中

## 3. 実現可能性(Feasibility)— 75/100

### 3.1 Critical Gaps(2件)

#### F-C1: 38エージェントの維持コストが過小評価

**現状**: 「v0.1 終了後にエージェント統廃合の振返り実施」と書かれているが、執筆段階で既に懸念されている。

**問題**: apex は12エージェント、WMAO は10エージェントで運用している。digital-product-studio-ai がいきなり38エージェントを抱えると、プロンプト管理・整合性維持・改訂コストが apex/WMAO 合算より重くなる。

**v0.2 改訂提案**: **段階的展開戦略を v0.2 で明文化**

```
v0.2 (Production-ready core):  20エージェント
   - Tier 0: 1
   - Tier 1: 5(全 Practice Director)
   - Tier 2: 6(主要 Discipline Lead のみ)
   - Tier 3: 6(Critical specialists のみ)
   - Tech Stack: 2(nextjs / wordpress のみ)

v0.3 (Expansion):              30エージェント(SaaS開発系を追加)
v0.4 (Full):                   38+エージェント(EC・i18n等の追加)
```

各段階で1案件回して検証してから次に進む。

**優先度**: Critical / **工数**: 大(設計再構築)

---

#### F-C2: マルチテナント並列5-10案件の現実的シミュレーション欠落

**現状**: マルチテナント設計はあるが、「5案件を並列に走らせた時の context window 上限」「同時実行されるエージェントの干渉」がシミュレーションされていない。

**問題**: 現実的には Claude のコンテキスト制約上、5案件並列はメモリ的に困難。1案件あたりに必要な context size の見積もりが要件にない。

**v0.2 改訂提案**: 
- 「同時アクティブ案件数の上限」を明記(現実的には2-3案件)
- 案件切替時のセッション分離プロトコルを追加
- `studio-director` に「案件切替時の context flush」責任を明記

**優先度**: Critical / **工数**: 中

---

#### F-H2: Hooks 7個の bash 実装が Windows 環境で動くか未検証

## 9. オープンクエスチョン(Shin への確認事項)

v0.2 着手前に Shin の判断が必要な項目:

### Q1: 段階展開戦略の採否

F-C1 で提案した「v0.2: 20エージェント → v0.3: 30 → v0.4: 38+」の段階展開を採用するか、それとも v0.2 で一気に38エージェントを完成させるか。

### Q2: 自社プロダクト開発を対象に含めるか

D-H3 で問うた点。受託専用にするか、AILEAP 自社プロダクトも対象に含めるか。

**Claude の推奨**: 含める。自社プロダクトでドッグフーディングすることで、digital-product-studio-ai 自体の品質が上がる。

### Q3: 動作環境の正式決定

F-H2 で問うた点。Windows のみ?Mac も?WSL2 必須?

**Claude の推奨**: Shin の作業環境(Windows + WSL2 が memory から推測)を v0.2 のプライマリ環境とし、Mac/Linux は best-effort 対応。

### Q4: 法務テンプレートの取扱い

F-H3 で問うた点。法務3テンプレート(プライバシーポリシー・利用規約・特商法)は v0.1 でひな型まで作るか、v0.2 以降に先送りするか。

**Claude の推奨**: ひな型は v0.2 で作成。ただし「※弁護士確認必須」のヘッダー強制挿入で責任分界を明確化。

### Q5: 多言語対応の前倒し可否

H-6 で問うた点。Shin の多言語スキルを差別化要素にするなら v0.2 で組み込む価値が高い。

**Claude の推奨**: 前倒し採用。日英中韓の4言語を標準対応とし、`localization-specialist` を v0.2 に追加。

### Q6: AILEAP 3組織を1リポジトリに統合するか

要件定義書 Section 17.2 で未解決として記載された点。

**Claude の推奨**: **別リポジトリ維持**。理由は (a) 各組織は独立した名前空間で管理する方が衝突がない (b) Shin が apex/WMAO で既に別リポジトリ運用に慣れている (c) 3組織を統合すると単一リポジトリが巨大化し Claude Code の性能が落ちる。
